# mortgagemath Review and Suggestions

This document was generated by a team of specialized AI agents tasked with reviewing the `mortgagemath` package, comparing it to other Python mortgage libraries, and identifying opportunities for expanding international and alternative mortgage coverage.

## 1. Senior Code and Architecture Review

**Overall Impression:** The codebase is extremely high-quality. The rigorous use of `decimal.localcontext(prec=50)` ensures precision and prevents ambient context changes from affecting calculations. The separation into `ROUND_EACH` and `CARRY_PRECISION` modes is an excellent touch for accommodating both bank statements and financial theory.

**Observations & Constructive Feedback:**
* **Interest-Only Loans:** There isn't an explicit way to handle interest-only (IO) periods. While you could simulate a non-amortizing loan using `payment_override = principal * periodic_rate`, many modern mortgages have an "Interest Only for X years, then amortizing for Y years" structure. Consider adding an `interest_only_months` parameter to `LoanParams`.
* **Zero Interest Mortgages:** `periodic_payment` correctly raises an error for 0% interest loans. However, 0% financing does exist (e.g., Islamic finance or promotional loans). Returning `principal / term_months` rounded appropriately would be a graceful way to support these rather than leaving it to the user.
* **Typing/Enums Validation:** In `LoanParams.__post_init__`, there are checks for rate schedule and terms, but one edge case is `payment_frequency=ANNUAL` with `compounding=MONTHLY`. While the math in `_periodic_rate_for` works out for most combinations, it might be worth double-checking if there are any financially invalid combinations of cadence/compounding that should raise a `ValueError`.
* **Data Science Integration:** The library returns a list of dataclasses. In the Python ecosystem, users analyzing mortgages will likely want this in a `pandas.DataFrame`. Providing a `to_pandas()` helper, or documenting `pd.DataFrame([vars(row) for row in sched])`, would be highly valuable.

## 2. Competitive Landscape & Presentation

**Competitor Analysis:**
* `numpy-financial` (`np.pmt`): Ubiquitous, but relies entirely on floats. Cannot match actual bank statements.
* `mortgage` (PyPI): Uses basic Decimals, but only supports simple US fixed-rate mortgages. No ARMs, no 30/360 vs Actual/360, no Canadian compounding.
* `amortization` (PyPI): Simple float-based tables.

**Market Positioning:** 
`mortgagemath` is objectively the most rigorous and cent-accurate library available in Python.

**Presentation Suggestions:**
* **Pandas/Matplotlib Examples:** Add examples to the documentation showing how to plot the amortization schedule (principal vs interest over time) using `matplotlib` or `plotly`. Data visualization is key to making the library "sticky" for data scientists.
* **Exporting:** Show how to export to Excel/CSV natively or via pandas. 
* **CLI Expansion:** The CLI (`mortgagemath schedule`) is great. Consider adding a summary command (`mortgagemath summary`) that outputs total interest paid, final payoff date, and APR.
* **Rename / Branding:** The branding is great (the logo and clear focus on cent-accuracy). Make sure to emphasize "Zero runtime dependencies" prominently in PyPI tags.

## 3. Global Mortgage Types & Feature Research

To become the definitive global standard for mortgage calculation, `mortgagemath` should expand its scope to include standard loan types outside the US and Canada.

### Identified Gaps in Repayment Structures
* **Constant Principal (Serial) Loans:** Common in **Norway, Sweden, and Denmark**. In a "Serielån", the borrower pays a fixed amount of principal each month plus the accrued interest. As a result, the total monthly payment decreases over time. This requires a new schedule path, distinct from the closed-form annuity formula.
* **Accelerated Bi-Weekly:** Very common in **Canada**. Unlike standard bi-weekly (which solves the annuity formula for 26 periods), "Accelerated" takes the *monthly* payment, divides it by 2, and charges that amount 26 times a year. This results in 1 extra monthly payment per year, paying the loan off faster. This can currently be hacked using `payment_override`, but a native constructor would be appreciated.
* **Offset Mortgages:** Common in the **UK and Australia**. The borrower's savings account balance is subtracted from the principal before daily/monthly interest is calculated. (This would require allowing a variable principal balance/schedule to be passed in, which may overcomplicate the current pure math approach).
* **Bausparvertrag / "Bauspardarlehen" (Germany/Austria):** A complex product where a savings phase transitions into a fixed low-interest loan phase. `mortgagemath` could handle the loan phase easily, but European users might expect a constructor.

### High-Quality European Source Material for Validation
If you are looking to expand validation fixtures, consider looking for regulatory documents from:
1. **FCA (Financial Conduct Authority) UK:** Provides examples of Annual Percentage Rate of Charge (APRC) calculations using standard UK practices.
2. **Finansinspektionen (Sweden):** Look for their guidelines on "Amorteringskrav" (amortization requirements), which specify how serial loans or mixed loans must be calculated.
3. **BaFin / PAngV (Germany):** The "Preisangabenverordnung" dictates how the effective interest rate (Effektivzinssatz) is calculated, providing rigorous mathematical examples of standard German "Annuitätendarlehen" (Annuity loans).

### Summary of Recommended Additions
1. Support for **Constant Principal / Serial Loans** (Nordic regions).
2. Support for **Interest-Only Periods** (Global CRE and UK/EU retail).
3. A helper for **Accelerated Bi-weekly** payments (Canada).
4. Examples showing **Pandas integration** and graphing.
