# mortgagemath v0.3+ Design

Status: **approved 2026-04-29**.  v0.3.0 implementation is in flight on
the `v0.3.0-intl-fixed-rate` branch. ARM work (v0.4) is documented here
but explicitly out of scope for v0.3.

## 1. Where we are (v0.2.2)

- 21 cents-level fixtures across CFPB, Fannie Mae §1103, Geltner,
  Goldstein, OpenStax, Las Positas, MS State, plus synthetics. Sources
  span $563 to $25M, 2%–12%, 5mo–30yr.
- Three independent axes: `DayCount` (30/360, Actual/360),
  `BalanceTracking` (round-each, carry-precision), `PaymentRounding`
  (ROUND_UP, HALF_UP, HALF_EVEN).
- Native balloon via `amortization_period_months`. Validated against
  Fannie Mae §1103 Tier 2 SARM.
- Hard constraints (project memory):
  1. Only add a mode/option when it lets a *specific published
     example* match exactly. Reference-driven, not feature-driven.
  2. Every published value from a source must reproduce, or the
     fixture stays out. No partial fixtures.

## 2. v0.3 — international fixed-rate

Two new mechanics — non-monthly **compounding** and non-monthly
**payment cadence** — unlock four independently citable Canadian
fixtures and one UK fixture. ARMs split out into v0.4.

### 2.1 Canadian semi-annual compounding (`j_2`)

The Canadian *Interest Act* §6 requires a mortgage rate be quoted as
either annual or semi-annual compounding (not monthly). A Canadian
"5% mortgage" with monthly payments uses the periodic rate
`(1 + 0.05/2)^(1/6) - 1 ≈ 0.41239...%`, **not** `0.05/12`. The
library's existing `r = annual / 12` cannot match any Canadian source.

New API:

```python
class Compounding(Enum):
    MONTHLY = "monthly"          # default — unchanged
    SEMI_ANNUAL = "semi_annual"  # Canadian Interest Act §6
    ANNUAL = "annual"            # for completeness
```

Adds `compounding: Compounding = Compounding.MONTHLY` to
`LoanParams`. Periodic-rate computation centralizes in a new helper:

```
MONTHLY      :  periodic = annual / (100 * payments_per_year)
                (preserves existing behavior exactly when frequency=MONTHLY)
SEMI_ANNUAL  :  periodic = (1 + annual/200)^(2/payments_per_year) - 1
ANNUAL       :  periodic = (1 + annual/100)^(1/payments_per_year) - 1
```

Backward compatible: existing fixtures unaffected.

### 2.2 Non-monthly payment cadence

```python
class PaymentFrequency(Enum):
    MONTHLY      = "monthly"       # 12/yr — default
    SEMI_MONTHLY = "semi_monthly"  # 24/yr
    BIWEEKLY     = "biweekly"      # 26/yr
    WEEKLY       = "weekly"        # 52/yr
    QUARTERLY    = "quarterly"     # 4/yr
    ANNUAL       = "annual"        # 1/yr
```

Adds `payment_frequency: PaymentFrequency = PaymentFrequency.MONTHLY`
to `LoanParams`.

`term_months` keeps its name (calendar duration in months).
Number of payments derives from
`term_months × payments_per_year / 12` and must come out to a whole
number (validated at construction).

### 2.3 Function rename

The existing `monthly_payment(loan)` function becomes
`periodic_payment(loan)`. `monthly_payment` stays as a permanent
alias so existing imports keep working — no deprecation.

`Installment.payment` already names the field correctly.
`[expected.monthly_payment]` in fixtures grows a synonym
`[expected.periodic_payment]`; both accepted, the new name preferred
in fresh fixtures.

### 2.4 Fixture format extensions (TOML)

```toml
[loan]
compounding = "semi_annual"          # default "monthly"
payment_frequency = "weekly"          # default "monthly"

[expected]
periodic_payment = "..."              # alias for monthly_payment
total_payable = "425891.00"           # optional — fee-inclusive total
total_interest = "233091.00"          # optional — total cost of credit
fees = "335.00"                       # optional — origination/booking
```

All new keys are optional and backward compatible.

### 2.6 Validation matrix and constraints

- `DayCount.ACTUAL_360` requires `Compounding.MONTHLY` and
  `PaymentFrequency.MONTHLY` (preserves all existing §1103 behavior;
  day-counted accrual is undefined under non-monthly cadence).
- `term_months × payments_per_year` must be divisible by 12. Rejected
  at `LoanParams` construction with a helpful error.
- Monthly + monthly is the existing fast-path (no exponentiation);
  identical numerical output to v0.2.x.

## 3. v0.4 (NOT IN v0.3) — ARMs

Two strong sources from the agent canvass. Documented for posterity;
no v0.3 implementation work.

### 3.1 Tier 1 — explicit rate schedule

```python
@dataclass(frozen=True)
class RateChange:
    effective_payment_number: int  # 1-indexed
    new_annual_rate: Decimal
    recast: bool = True            # True = recompute payment over remaining term

@dataclass(frozen=True)
class LoanParams:
    ...
    rate_schedule: tuple[RateChange, ...] = ()
```

Validates Goldstein 12e §10.4 Example 13 (5/1 ARM, $200k / 5.7% →
7.2% → 9.6% trajectory).

### 3.2 Tier 2 — index + caps

For Reg Z Sample H-14 (12 CFR Part 1026 Appendix H): 1/1 ARM with
2 pp annual cap and 5 pp lifetime cap, 15-year historical index
trajectory. Add `IndexedRateSchedule` carrying index history, margin,
periodic cap, lifetime cap, floor.

### 3.3 Deferred — payment cap with negative amortization

Goldstein Example 14 is the only published source we found.
Per the two-source rule (analogous to `BalanceTracking.CARRY_PRECISION`
waiting for Geltner + Goldstein), defer until a second source emerges.

## 4. Beyond v0.4

- **Australian Actual/365**: API sketched (`DayCount.ACTUAL_365`),
  no published cents-level worked example with daily accruals + payment
  trajectory found. Defer until a fixture exists.
- **Republic of Ireland**: lender disclosures diverge €0.75–€1.12
  with no documented day-count convention. Defer until a primary
  source publishes the algorithm.
- **UK reversion-to-SVR multi-rate**: needs the Tier-1 ARM rate
  schedule plus a UK-specific verifiable example. None found in the
  v0.3 canvass.
- **Custom whole-dollar payment override**: needed for Olivier
  Maksim ($2,260) and Kerry's Toyota ($2,691). Cheap to add when a
  second source justifies it.
- **Scotland-specific**: no math difference from England/Wales.
  Documentation note only.

## 6. Implementation plan (v0.3.0 only)

1. `src/mortgagemath/_types.py` — add `Compounding`, `PaymentFrequency`;
   add fields and validation to `LoanParams`.
2. `src/mortgagemath/_payment.py` — refactor closed-form annuity to
   use a periodic-rate helper. Rename `monthly_payment` →
   `periodic_payment`; add `monthly_payment` alias.
3. `src/mortgagemath/_schedule.py` — generalize 30/360 round-each and
   carry-precision paths to use periodic rate × number-of-payments.
   Restrict ACTUAL_360 to monthly + monthly.
4. `src/mortgagemath/__init__.py` — export `Compounding`,
   `PaymentFrequency`, `periodic_payment`. Keep `monthly_payment`.
5. `src/mortgagemath/__main__.py` — no behavioral change; `monthly_payment`
   alias keeps working.
6. `tests/conftest.py` and `tests/test_bank_schedules.py` — fixture
   loader accepts `compounding`, `payment_frequency`,
   `periodic_payment`, and the new total/fees keys.
7. `tests/test_payment.py`, `tests/test_schedule.py` — add tests for
   non-monthly compounding & frequency, plus `monthly_payment` alias.
8. Add 6 fixtures (5 Canadian + 1 UK) to `tests/schedules/`.
9. Update `pyproject.toml` to `0.3.0`.
10. Update `CHANGELOG.md`, `README.md`, `docs/accuracy.md`,
