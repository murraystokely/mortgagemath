# mortgagemath v0.4.0 — ARMs and CLI

Status: **implemented, awaiting commit and release.**  Branch:
`v0.4.0-arms-and-cli`.

The release ships Tier 1 ARMs (explicit ``rate_schedule`` of
``RateChange`` entries) plus a first-class CLI.  Tier 2
(``IndexedRateSchedule`` with library-side cap derivation) was
considered but deferred — Reg Z Sample H-14 reproduces every
published anchor to the cent using Tier 1 alone with hand-computed
post-cap rates encoded in the fixture, and only one source
currently motivates Tier 2.  When a second cap-using source
surfaces, that helper becomes justified.

This release ships two coordinated additions:

1. **Tier 1 ARM support** — explicit per-period rate-change schedule
   (`RateChange` dataclass + `LoanParams.rate_schedule` tuple).
   Validates Goldstein 12e §10.4 Example 13 (years 1–6 of a $200k 5/1
   ARM).
2. **A first-class command-line interface** — `mortgagemath payment`,
   `mortgagemath schedule`, `mortgagemath selfcheck`.  Stdlib only
   (argparse).  Accepts ARMs.

Tier 2 ARMs (index history + periodic + lifetime caps, validating Reg
Z Sample H-14) and payment-cap / negative-amortization (Goldstein
Example 14) stay deferred — see §9 below.

## 1. ARM API design

### 1.1 `RateChange` dataclass (new)

```python
@dataclass(frozen=True, slots=True)
class RateChange:
    effective_payment_number: int   # required, 1-indexed payment ordinal
    new_annual_rate: Decimal        # required, percentage (e.g. Decimal("7.2"))
    recast: bool = True             # default True; False = keep prior payment
```

- `effective_payment_number` is **1-indexed** and means: starting with
  the payment whose `Installment.number == effective_payment_number`,
  interest accrues at `new_annual_rate`.  Goldstein's "Year 6 begins
  at month 61" maps to `effective_payment_number=61`; month 61 interest
  is `185,405.12 * 0.072/12 = 1,112.43`, matching the textbook.
- `new_annual_rate` is a `Decimal` percentage to mirror
  `LoanParams.annual_rate`.  Required, must be `> 0`.
- `recast=True` (the default) recomputes the level payment for the
  remaining payments at the new rate.  `recast=False` keeps the prior
  payment (used for IO periods or step-up loans where the borrower's
  payment is fixed by contract).  v0.4.0 only ships fixtures
  exercising `recast=True`, but the field is on the dataclass from
  day one because it's free.

No `effective_date`.  Date math is irrelevant for 30/360 schedules;
everything keys off payment ordinal.  If/when an Actual/360 ARM
fixture surfaces we revisit.

### 1.2 `LoanParams` field (new)

Add exactly one field at the end of the dataclass to preserve
positional-arg compatibility:

```python
rate_schedule: tuple[RateChange, ...] = ()
```

Empty tuple is the default and triggers the v0.3.0 fast path with
byte-identical output.  Every existing fixture passes unchanged.

### 1.3 Validation (in `LoanParams.__post_init__`)

1. `effective_payment_number >= 2` for every entry.  Payment 1 with a
   different rate is the same as constructing the loan with that
   rate — reject as a footgun.
2. `effective_payment_number <= total_payments`.  A rate change after
   the last payment is dead code; reject.
3. `effective_payment_number` strictly increasing across the tuple.
4. `new_annual_rate > 0` for each entry.
5. `rate_schedule` non-empty implies `day_count == THIRTY_360`.  ARM
   + Actual/360 stays unimplemented in v0.4.0; relax when a fixture
   motivates it.
6. `rate_schedule` non-empty implies
   `amortization_period_months == term_months` (no balloon ARMs in
   v0.4.0).  Relax when a fixture motivates it.
7. `rate_schedule` is allowed under both `BalanceTracking` modes; the
   recast formula adapts (see §2).

### 1.4 Recast numerical recipe

When a `RateChange` with `recast=True` fires at payment `k`:

- Let `B` = balance carried into period `k` *before* interest accrual
  for period `k`.  Under `ROUND_EACH` this is the rounded balance from
  row `k-1`.  Under `CARRY_PRECISION` it is the unrounded internal
  balance.
- Let `r_new` = the periodic rate for `new_annual_rate` under the
  loan's `compounding` and `payment_frequency`.  Cleanest
  implementation: factor out a
  `_periodic_rate_for(annual_rate, compounding, payment_frequency)`
  helper and have the existing `_periodic_rate(loan)` delegate.
- Let `m` = remaining payments = `total_payments - (k - 1)`.
- New level payment (carry-precision arithmetic, then quantize to
  cents using `loan.payment_rounding`):

```
factor = (1 + r_new) ** m
pmt_new = B * r_new * factor / (factor - 1)
pmt_new_disp = pmt_new.quantize(_PENNY, rounding=loan.payment_rounding)
```

Verification against Goldstein: `B = 185405.12`, `r_new = 0.006`,
`m = 300`.  `(1.006)^300 ≈ 6.022575`,
`185405.12 × 0.006 × 6.022575 / 5.022575 ≈ 1334.151`, rounds
HALF_UP to **$1,334.15** — matches exactly.

For `recast=False`, the level payment carries through unchanged; only
the periodic rate updates from period `k` onward.

## 2. ARM schedule generation — algorithmic change

Both `_schedule_thirty_360_round_each` and
`_schedule_thirty_360_carry_precision` grow the same shape: a
per-iteration check against the `rate_schedule` tuple.  The
Actual/360 path is unchanged (validation forbids `rate_schedule`
there).

### 2.1 Pseudocode (round-each path)

```
schedule_idx = 0
periodic_rate = _periodic_rate(loan)
pmt = periodic_payment(loan)

for i in 1..total_payments:
    # Step 1: apply any rate change effective AT this payment.
    if (schedule_idx < len(rate_schedule)
            and rate_schedule[schedule_idx].effective_payment_number == i):
        rc = rate_schedule[schedule_idx]
        periodic_rate = _periodic_rate_for(
            rc.new_annual_rate, loan.compounding, loan.payment_frequency
        )
        if rc.recast:
            remaining = total_payments - (i - 1)
            pmt = recast_payment(B=balance, r=periodic_rate, m=remaining,
                                 rounding=loan.payment_rounding)
        schedule_idx += 1

    # Step 2: existing per-row body, using current periodic_rate and pmt.
    interest = (balance * periodic_rate).quantize(_PENNY, rounding=interest_rounding)
    ...
```

### 2.2 Carry-precision path

Identical control flow, but the recast uses the unrounded `balance`
Decimal already maintained in that loop, and both `pmt_disp` and
`pmt_raw` update on recast.

### 2.3 Boundary semantics

The new rate applies *on* the period whose
`number == effective_payment_number`.  This is the only ordering that
matches Goldstein.

### 2.4 Final-row trueup interaction

The existing final-row "land balance at exactly zero" logic in both
paths is unaffected.  A recast at any earlier payment changes the
level payment going forward but the final-row trueup still runs at
payment `total_payments`.  The `will_pay_off_early` heuristic in
round-each must compare against the *current* (post-recast) `pmt`,
which the pseudocode above already gives because `pmt` is mutated in
place.

### 2.5 Empty `rate_schedule` is the fast path

With an empty tuple, the per-iter cost is one int comparison.  More
importantly, every existing fixture must produce byte-identical
output — enforced by a regression test that reruns the v0.3.0 fixture
suite unchanged.

## 3. Goldstein Example 13 — investigated, not shipped

Implementation discovery: Goldstein 12e §10.4 Example 13 publishes
checkpoint balances computed via the textbook closed-form
outstanding-balance formula ``pmt × (1 - (1+r)^-(n-k))/r`` using the
rounded payment $1,160.80.  The library's row-by-row schedule
produces a balance at month 60 of $185,405.25 (carry-precision) or
$185,405.38 (round-each), differing from Goldstein's $185,405.12 by
13–26¢.  The recast at month 61 therefore yields $1,334.16, not
Goldstein's $1,334.15.

This is an algorithmic difference, not a library bug — Goldstein
computes balances at conceptual checkpoints, while the library
produces actual amortization rows.  Adding a "PV-style balance"
mode purely to chase Goldstein would violate the project's
complexity-threshold rule (one source).  The fixture is **not
committed**; the situation is documented in ``CHANGELOG.md`` and
``tests/test_arms.py``.

## 3b. Reg Z Sample H-14 — shipped as the canonical published-source ARM fixture

```python
LoanParams(
    principal=Decimal("10000"),
    annual_rate=Decimal("17.41"),
    term_months=360,
    payment_rounding=PaymentRounding.ROUND_HALF_UP,
    interest_rounding=PaymentRounding.ROUND_HALF_UP,
    balance_tracking=BalanceTracking.ROUND_EACH,
    rate_schedule=tuple(
        RateChange(effective_payment_number=12 * (yr - 1) + 1,
                   new_annual_rate=Decimal(post_cap_rate))
        for yr, post_cap_rate in enumerate(
            ["15.41", "15.17", "13.17", "12.41",
             *(["12.41"] * 10)],   # years 6–15: lifetime cap binds
            start=2,
        )
    ),
)
```

**Published values reproduced exactly** (regulation publishes year-end
snapshots for each of 15 years):

- Year 1 P&I: **$145.90**.
- Year-end balances (years 1–5): **$9,989.37**, **$9,969.66**,
  **$9,945.51**, **$9,903.70**, **$9,848.94**.
- Year-15 terminal balance: **$8,700.37**.

The fixture CSV ships first-of-year + last-of-year anchor rows for
years 1–5 (10 rows) plus the year-15 terminal balance (1 row) — 11
schedule rows total.  Reg Z's published periodic cap binds at years
2 and 4; the symmetric lifetime cap binds from year 5 through year
15 (where the rate stays at 12.41% but the payment recasts annually
on the new remaining balance).

## 4. CLI design

A first-class CLI accessible via both `mortgagemath …` and
`python -m mortgagemath …`.  **Stdlib only — no new runtime
dependencies.**  argparse with subcommand dispatch.

### 4.1 Why argparse, not click / typer

The library is intentionally zero-runtime-dep.  The community
convention for *libraries* (vs. *tools*) that ship a CLI is argparse:
`pip` and `pipx` use stdlib argparse; click is the convention for
standalone tools (`black`, `pip-tools`).  argparse covers our ~10-flag
surface in roughly 80 lines of code.  Adding click behind a `[cli]`
extras group is defensible (precedent: `httpx[cli]`) but overkill for
the surface we need.

### 4.2 Console-script entry point

Add to `pyproject.toml`:

```toml
[project.scripts]
mortgagemath = "mortgagemath.__main__:main"
```

Both `mortgagemath …` and `python -m mortgagemath …` route through
the same `main()`.  This is the convention used by `pip`, `black`,
`pytest`, `mypy`, `build`, `twine`, `venv`.

### 4.3 Subcommands

```
mortgagemath payment    [LoanParams flags]
mortgagemath schedule   [LoanParams flags] [--format table|csv|json]
mortgagemath selfcheck  (existing post-install validation)
```

If invoked with no subcommand, default to `selfcheck` — preserves
the existing `python -m mortgagemath` behavior (and the
`tests/test_selfcheck.py` contract).

### 4.4 Flags

Match `LoanParams` field names exactly with `--kebab-case`.  Choice
values use the enum's `.value` string verbatim, so the CLI is
self-documenting against the library reference:

| Library field | Flag | Type / choices |
|---|---|---|
| `principal` | `--principal`, `-p` | Decimal string |
| `annual_rate` | `--rate`, `-r` | Decimal string |
| `term_months` | `--term-months`, `-t` | int |
| `day_count` | `--day-count` | `30/360`, `actual/360` |
| `payment_rounding` | `--payment-rounding` | `ROUND_UP`, `ROUND_HALF_UP`, `ROUND_HALF_EVEN` |
| `interest_rounding` | `--interest-rounding` | (same) |
| `start_date` | `--start-date` | `YYYY-MM-DD` |
| `amortization_period_months` | `--amortization-period-months` | int |
| `balance_tracking` | `--balance-tracking` | `round_each`, `carry_precision` |
| `compounding` | `--compounding` | `monthly`, `semi_annual`, `annual` |
| `payment_frequency` | `--payment-frequency` | `monthly`, `semi_monthly`, `biweekly`, `weekly`, `quarterly`, `annual` |
| `rate_schedule` | `--rate-change` (repeatable) | `EFFECTIVE_PMT:NEW_RATE[:no_recast]` |

Short flags only for the three universal ones (`-p / -r / -t`).
Long-tail options use long flags only.

### 4.5 ARM support in the CLI

`--rate-change` is repeatable (argparse `action="append"`).  Format:

```
EFFECTIVE_PMT:NEW_RATE
EFFECTIVE_PMT:NEW_RATE:no_recast
```

`recast=True` is the default; an explicit `:no_recast` suffix sets it
to False.  Examples:

```sh
# Goldstein 5/1 ARM, year 6 reset
mortgagemath schedule --principal 200000 --rate 5.7 --term-months 360 \
    --payment-rounding ROUND_HALF_UP --interest-rounding ROUND_HALF_UP \
    --rate-change 61:7.2

# Multiple rate changes (synthetic)
mortgagemath payment --principal 100000 --rate 5 --term-months 360 \
    --rate-change 61:6 --rate-change 121:7

# Step-up loan with no recast (rate moves but payment stays flat)
mortgagemath schedule --principal 100000 --rate 5 --term-months 360 \
    --rate-change 61:6:no_recast
```

### 4.6 Output formatting

- **`--format table`** (default): right-aligned columns with
  comma-grouping (`f"{x:>12,.2f}"`).  Stdlib only.
- **`--format csv`**: `csv.writer(sys.stdout).writerow(...)`.  Stdlib.
- **`--format json`**: `json.dumps([...], default=str)`.  Decimal
  values serialize as **strings** to preserve cents (downstream tools
  that don't know about Decimal would otherwise lossily coerce).
- The `payment` subcommand has no `--format`; it just prints the
  Decimal periodic payment to stdout, one line, for clean piping.

### 4.7 Sketch of `__main__.py`

```python
def main(argv: list[str] | None = None) -> int:
    parser = argparse.ArgumentParser(prog="mortgagemath")
    sub = parser.add_subparsers(dest="cmd")

    sub.add_parser("selfcheck", help="Run post-install reference checks")

    common = argparse.ArgumentParser(add_help=False)
    common.add_argument("--principal", "-p", type=Decimal, required=True)
    common.add_argument("--rate", "-r", type=Decimal, required=True,
                        dest="annual_rate")
    common.add_argument("--term-months", "-t", type=int, required=True)
    common.add_argument("--day-count",
                        choices=[e.value for e in DayCount],
                        default=DayCount.THIRTY_360.value)
    # ... other LoanParams fields, all optional with library defaults
    common.add_argument("--rate-change", action="append", default=[],
                        type=_parse_rate_change,
                        help="EFFECTIVE_PMT:NEW_RATE[:no_recast] (repeatable)")

    sub.add_parser("payment", parents=[common])
    p_sched = sub.add_parser("schedule", parents=[common])
    p_sched.add_argument("--format", choices=["table", "csv", "json"],
                         default="table")

    args = parser.parse_args(argv)
    if args.cmd in (None, "selfcheck"):
        return _run_selfcheck()
    params = _params_from_args(args)
    if args.cmd == "payment":
        print(periodic_payment(params))
        return 0
    return _emit_schedule(params, args.format)
```

`_parse_rate_change`, `_params_from_args`, `_emit_schedule` are
~20-line helpers each.

## 5. Tests required

### 5.1 ARM unit tests (`tests/test_arms.py`, new)

- `RateChange.__post_init__` (or library-level) validation:
  `effective_payment_number=0` rejected, `=1` rejected, non-positive
  `new_annual_rate` rejected.
- `LoanParams.__post_init__` validation: non-strictly-increasing
  schedule rejected; entry past `total_payments` rejected;
  `rate_schedule` non-empty with `day_count=ACTUAL_360` rejected;
  `rate_schedule` non-empty with `amortization_period_months !=
  term_months` rejected.

### 5.2 ARM schedule tests (`tests/test_schedule.py` additions)

- Synthetic 5y / 360mo loan with rate doubling at month 60, both
  `ROUND_EACH` and `CARRY_PRECISION`.  Assert `principal + interest
  == payment` for every row, final balance `== Decimal("0.00")`,
  interest at row 60 reflects the new rate.
- `recast=False` test: rate changes, payment stays equal to original,
  balance still lands at zero (final-row trueup absorbs the
  difference).
- Multiple rate changes on consecutive periods (synthetic stress).
- Empty `rate_schedule` produces a schedule byte-identical to the
  same `LoanParams` constructed without the field.

### 5.3 ARM fixture

`tests/schedules/goldstein_ch10_4_ex13_5y1arm_200k_57_360mo.{toml,csv}`,
loaded by the existing `test_bank_schedules.py` machinery.  The
fixture loader (`_loan_from_toml` in `test_bank_schedules.py`)
extends to parse `[[loan.rate_schedule]]`.

### 5.4 CLI tests (`tests/test_cli.py`, new)

- `mortgagemath` with no args runs selfcheck, exits 0.
- `mortgagemath selfcheck` exits 0.
- `mortgagemath payment --principal 131250 --rate 4.25 --term-months 360`
  prints exactly `645.68\n`.
- `mortgagemath schedule … --format csv` produces parseable CSV with
  the right header.
- `mortgagemath schedule … --format json` produces valid JSON that
  parses, and Decimal values are strings.
- `mortgagemath schedule --rate-change 61:7.2 …` produces a schedule
  whose post-month-60 payments reflect the recast.
- Bad `--rate-change` syntax (e.g., `61` without rate) exits 2 with
  argparse's standard error message.

### 5.5 Self-check (`__main__.py`)

The existing self-check stays tight; do **not** add ARM probes.

## 8. Risks and edge cases

- **Rate change on the last payment.**  Validation rejects
  `effective_payment_number > total_payments`; `==` is allowed.
  With `recast=True` and `m = 1`, the recast formula reduces to
  `pmt = B * (1 + r_new)`, which the final-row trueup overrides
  anyway.  Add a test; document.
- **Balloon loans + ARM** (`amortization_period_months > term_months`):
  rejected by validation in v0.4.0.  Reconsider when a fixture
  surfaces.  Document.
- **ACTUAL_360 + `rate_schedule`**: rejected by validation.
  Document.
- **Multiple consecutive rate changes**: allowed; covered by the
  stress test in §5.2.
- **Empty `rate_schedule` byte-identical with v0.3.0**: enforced by
  the regression test running every existing fixture.
- **`recast=False` interaction with final-row trueup**: when the
  prior payment is held flat across a rate change, the schedule may
  end with a non-trivial residual that the trueup absorbs.  Test
  `principal + interest == payment` and final balance zero;
  document that final-row payment may differ noticeably from the
  level payment under `recast=False`.
- **Negative-rate sanity**: validation rejects non-positive
  `new_annual_rate`.  Same treatment as `LoanParams.annual_rate`.
- **CLI `--format json` Decimal serialization**: Decimal serializes
  as a string (`json.dumps(..., default=str)`).  Tests verify the
  output is valid JSON and that downstream consumers can recover
  exact Decimal values from the strings.
- **CLI default subcommand**: `mortgagemath` (no args) routes to
  `selfcheck`.  `tests/test_selfcheck.py` continues to assert this.
- **Backward compat for `monthly_payment` alias**: still works,
  unchanged.
