# mortgagemath v0.6.0 — Rare/historic conventions

Status: **planning.** Branch (when started): `v0.6.0-rare-conventions`.

## Scope

Two narrowly-scoped library extensions, each unlocking a specific
class of historic/regional published amortization tables that the
current closed-form-only library cannot reproduce. Both are
opt-in additions to ``LoanParams``; neither alters the behavior
of any existing fixture or default.

1. **`payment_override`** — let the user pin the periodic payment
   amount, with the schedule's final row absorbing whatever
   residual is left. Unlocks the "given-payment, find-term"
   convention used by the FHLBB *Federal Home Loan Bank Review*
   of March 1935 and similar pre-modern sources where the lender
   chose the payment first (typically a round percentage of
   original principal) and accepted whatever final-payment trueup
   the math produced.
2. **`fee_per_period`** — let a flat fee ride on top of the pure
   interest+principal payment, so `Installment.payment` reflects
   the *gross annuité* the borrower actually paid. Originally
   motivated by the 1852 Crédit Foncier de France structure
   (4½% interest + 1% amortissement + 0.5% combined frais /
   réserve / impôt = 6.00 fr. per 100 fr.) but also covers a
   class of modern French *tableaux d'amortissement* that publish
   only the gross *total échéance*.

## Rationale and trigger sources

### Feature 1: `payment_override`

The library has been historically blocked from including the
**FHLBB *Review*, March 1935, "Direct-Reduction Plan A"** schedule
(*$3,000 / 6% / monthly compounding / $30 monthly payment, 138
full payments + 139th of $29.27 = full retirement*). The closed-
form payment for `term_months=139` is $29.9964, which rounds to
$30.00, and the library's current trueup logic produces a final
row of $29.35 — eight cents off the published $29.27.

The eight-cent gap is **structural, not a rounding bug**. FHLBB
chose $30 (= 1% of original principal) as the payment by
convention and let the term and trueup follow; the library's
modern convention chooses an integer term and derives a level
payment. The simulation below confirms that with a
`payment_override = Decimal("30.00")` field and carry-precision
balance tracking, the library reproduces every FHLBB cell
exactly, including $29.27 on row 139:

```
month   1: payment=30.00 interest=15.00 principal=15.00 balance=2985.00
month   2: payment=30.00 interest=14.93 principal=15.07 balance=2969.92
...
month 138: payment=30.00 interest= 0.29 principal=29.71 balance=  29.13
month 139: payment=29.27 interest= 0.15 principal=29.12 balance=   0.00
```

This pre-validates the design *before* writing library code — the
v0.6.0 trigger condition (a verifiable published source we can
match cell-for-cell) is satisfied.

The same feature would also accept a class of pre-1968 American
sources that share the "round-payment, accept-trueup" convention.
It is not on the immediate fixture target list, but the FHLBB
1935 fixture alone justifies the feature under the
complexity-threshold rule.

### Feature 2: `fee_per_period`

The 1852 CF rate decomposition (Wolowski 1852) explains why every
canonical 19th-century French amortization table publishes a
*gross annuité* that the library's pure interest+principal model
cannot reproduce. The proposed fee field treats the fee as a
flat amount added to each period's `Installment.payment`, with
the closed-form payment derivation untouched.

A v0.6.0 sub-agent canvass of modern French residential mortgage
practice produced the following scoping verdict:

- The **canonical modern French *Tableau d'amortissement*** (per
  Crédit Agricole, Meilleurtaux, BNP Paribas, ANIL, and
  service-public.fr templates) publishes **separate columns** for
  *capital amorti*, *intérêts*, *assurance emprunteur*, and
  *total échéance*. The library's existing closed-form already
  reproduces the *capital + intérêts* column at standard nominal
  rates; no fee feature is required for the
  pure-amortization slice.
- **Assurance emprunteur** is overwhelmingly priced as
  *taux d'assurance × original principal*, paid as a flat amount
  per period. This is exactly the structure `fee_per_period`
  models. Sources that publish the *total échéance* column (or
  the older CF-style combined annuity) would become matchable
  under `fee_per_period`.
- **What the simple flat-amount fee does NOT cover**:
  - "Assurance sur le capital restant dû" (insurance computed on
    declining balance) — produces a *declining* per-period fee,
    not flat. Available as an option from most French insurers,
    but the flat-original-principal structure remains dominant.
  - Stepped insurance schedules (rate by borrower age / LTV bucket).
  - Frais de garantie / caution paid as periodic installments at
    a different cadence than the loan itself.

  These three structures, if a published source ever motivates
  matching them, would require a richer fee parameter — a
  `FeeSchedule` data type analogous to `RateSchedule`. That is
  **out of scope for v0.6.0**.

The v0.6.0 trigger source for `fee_per_period` is more tentative
than for `payment_override`. The Wolowski 1852 essay publishes
only the rate decomposition (no row-by-row schedule), and the
row-level CF tables in Bellet 1854 and Josseau 1872 are behind
JS/captcha walls on Gallica that have resisted automated
retrieval. Under the complexity-threshold rule, **shipping
`fee_per_period` should be conditional on first retrieving one
verifiable CF-era or modern-French row-level table** that the
library can match cell-for-cell with the proposed semantics.

If that retrieval succeeds, both features ship in v0.6.0
together. If not, only `payment_override` (which has a confirmed
trigger) ships, and `fee_per_period` deferred to v0.6.1 or later
when a verified source surfaces.

## API design

### `LoanParams.payment_override`

```python
@dataclass(frozen=True)
class LoanParams:
    # ... all existing fields ...
    payment_override: Decimal | None = None
```

**Semantics:**

- When `None` (default), the library behaves as today: the
  closed-form payment is derived from `(principal, annual_rate,
  term_months)` and used as the level payment for all rows
  before the final-row trueup.
- When set to a `Decimal`, the library uses this value as the
  payment for every row except the final (or partial) one. The
  final row is computed as:
  `payment_final = remaining_balance + final_period_interest`,
  rounded once to cents using `payment_rounding`.
- If `payment_override` is *too small* to retire the loan in
  `term_months` (i.e. the loan would balloon), the final row
  payment includes the unpaid residual as principal — i.e. a
  balloon payment. If this exceeds a sanity threshold (e.g.
  10× the override), `LoanParams.__post_init__` raises a clear
  error: the user almost certainly meant a different combination
  of inputs.
- If `payment_override` is *too large* (the loan retires before
  `term_months`), the schedule has fewer rows than `term_months`
  and the library logs a warning.
- The override interacts with `rate_schedule`: when a rate change
  would normally trigger a recast of the level payment, the
  override *takes precedence* — the new rate's interest accrues
  but the payment stays at the override. This matches the FHLBB
  1935 convention. ARM payment-cap factors are silently ignored
  while the override is in force.
- `periodic_payment(loan)` returns the override directly when set;
  otherwise returns the closed-form value as today.

### `LoanParams.fee_per_period`

```python
@dataclass(frozen=True)
class LoanParams:
    # ... all existing fields ...
    fee_per_period: Decimal = Decimal("0")
```

```python
@dataclass(frozen=True)
class Installment:
    number: int
    payment: Decimal      # = interest + principal + fee  (gross échéance)
    interest: Decimal
    principal: Decimal
    fee: Decimal          # default 0.00; non-zero only for fee-loaded loans
    balance: Decimal
```

**Semantics:**

- When `fee_per_period == 0` (default), `Installment.fee == 0`
  and `Installment.payment == interest + principal` — identical
  to today's behavior. **All 35 existing fixtures pass unchanged.**
- When non-zero, the fee is added to *every* period's payment,
  including the final-row trueup (so the borrower's last echeance
  is also gross). The fee does not interact with interest or
  principal: closed-form payment derivation, rate-schedule recasts,
  and balance tracking are all computed at the
  interest+principal level only.
- The fee is treated as a **per-period flat amount in the loan's
  currency**, not a percentage. Sources that quote the loading as
  a percentage of original principal (CF: 0.50 fr. per 100 fr. per
  year; modern French *taux d'assurance*: 0.36% × original
  principal per year) require a one-line preprocessing step:
  `fee_per_period = original_principal * fee_pct / payments_per_year`.
- Validation: `fee_per_period >= 0`. Negative fees are not modeled.

### Test schema (`tests/schedules/<name>.toml`)

Two new optional `[loan]` keys, both Decimal strings:

```toml
[loan]
# ... existing fields ...
payment_override = "30.00"        # optional
fee_per_period   = "0.50"         # optional
```

Both default to omission (no behavior change). The `<name>.csv`
column structure for fee-loaded fixtures gains a `fee` column
between `interest` and `balance`:

```csv
payment,payment_amount,principal,interest,fee,balance
```

Existing CSVs without a `fee` column are accepted unchanged
(the loader treats the missing column as `Decimal("0.00")`).

## Validation

### v0.6.0 fixtures targeted

1. **FHLBB 1935 Plan A** — `fhlbb_1935_plan_a_3k_6pct_30mo_payment.toml`,
   confirmed matchable cell-for-cell with `payment_override`.
2. **FHLBB 1935 Plan B** — same source, semiannual interest credit,
   140 payments + 141st of $20.43. Will require `payment_override`
   plus configuration we already support.
3. **CF Wolowski 1852** — *if* a row-level schedule can be
   retrieved manually from Bellet, Josseau, or Pinschof. Pure
   `fee_per_period` test.
4. **Modern French *Tableau d'amortissement*** — *if* a published
   table with a *total échéance* column at cents precision can
   be located (ANIL or service-public.fr templates are the most
   likely candidates). Pure `fee_per_period` test.

### Non-fixture tests

- Unit tests covering `payment_override` interactions with each
  rate-schedule path (no rate change; fixed rate change with no
  recast; rate change with recast — override should suppress the
  recast).
- Unit tests covering `fee_per_period` with each `payment_frequency`
  and `compounding` combination (the fee should add cleanly
  without affecting the closed-form payment).
- Property test: for every existing fixture, setting
  `payment_override = periodic_payment(loan)` and
  `fee_per_period = Decimal("0")` should produce a schedule
  byte-identical to today's output. (Confirms the new fields are
  truly opt-in.)

## What's NOT in v0.6.0

- **Variable / outstanding-balance fees** (CF-style "intérêt aux
  porteurs" computed on declining balance, French
  "assurance sur le capital restant dû"). Would require a
  `FeeSchedule` data type. Defer until a published source forces
  the issue.
- **Stepped insurance schedules** (age-bucket or LTV-bucket
  insurance rates). Same reason.
- **Frais de garantie / caution at separate cadence**. Same reason.
- **A real "given-payment, find-term" mode** that returns the
  computed term as output. The override approach treats
  `term_months` as an upper bound and the library reports
  `actual_rows` after the fact; an explicit `find_term()`
  function is a separate API surface that no specific source
  motivates.
- **APR / TAEG validation**. The Reg Z Appendix J actuarial-method
  APR validator was deferred from v0.5; remains deferred. TAEG
  computation requires modeling all upfront and ongoing fees,
  which is a much larger scope than this release admits.

## Risk assessment

- **Backwards compatibility.** Both new fields are optional with
  sentinel defaults that produce the existing behavior. The
  Installment dataclass gains a field, which is technically
  source-incompatible for callers that destructure positionally,
  but `Installment` is conventionally accessed by attribute (and
  the existing test suite does so). Fixture CSVs without a `fee`
  column continue to load.
- **Validation depth.** The complexity-threshold rule has been
  satisfied for `payment_override` (FHLBB 1935 confirmed
  cell-for-cell pre-implementation). For `fee_per_period` the
  trigger is conditional on retrieving one row-level CF or modern
  French source; if none surfaces, ship `payment_override` alone
  and defer the fee feature.
- **Decimal arithmetic.** Both features stay strictly within the
  existing `Decimal` arithmetic paths. No new floating-point
  introductions.

## Out of scope (and why)

- **A general-purpose accounting framework for compound mortgage
  fees, taxes, and insurance.** The library is a closed-form
  amortization calculator with cents-level matching against
  published sources, not a TILA / HLPM disclosure builder. If
  the target use case requires modelling the full
  Closing-Disclosure / FISE European Standardised Information Sheet
  fee structure, a separate library is warranted.

between — outweighs it once the corpus passes ~30 fixtures. The
v0.6.0 release replaces them with a single comprehensive
**Examples** vignette, organized by country convention then by
loan type.

### `examples.qmd` — proposed structure

Each example block in the new vignette has the same five
elements, in this order:

1. **Scenario** (one-paragraph description of the borrower / loan / situation)
2. **Library configuration** — a `LoanParams(...)` literal with
   every relevant knob explicit
3. **Command-line invocation** — the `mortgagemath` CLI form
   that produces the same schedule
4. **Validated worked example** — published-source anchors the
   library matches to the cent, executed inline as a Python
   chunk so Quarto re-renders them on each build
5. **Source** — citation, URL, fixture filename for cross-reference

A representative subset (not exhaustive):

#### United States

  OpenStax §6.8 home-loan example, $136,700 / 5.75% / 15 yr,
  ROUND_UP (the textbook explicitly specifies ceiling rounding

  Fannie Mae Multifamily Selling and Servicing Guide §1103

Currently no fixtures; the section header reserves the slot.

### Migration mechanics

- Move the worked example, library config, CLI, and Python chunk
  from each of the three folded vignettes into `examples.qmd`
  under the appropriate country section, preserving the source
  citation and any narrative commentary.
- Delete the old vignette files outright (`arm-regz-h14.qmd`,
  `payment-caps-proeducate.qmd`, `canadian-j2.qmd`). They were
  not in place long enough for external links to matter.
- Update `_quarto.yml` navbar and `index.qmd` table to list the
  four-vignette set: at-a-glance, validation, examples, history.
- Cross-references in body text — currently the validation
  vignette mentions "the same six knobs documented in the
  *at-a-glance* vignette" and the history vignette refers
  forward to specific examples — are updated to point at the
  new consolidated locations.
- The three folded vignettes' rendered PDFs in
  `docs/vignettes/rendered/` are deleted in the same commit;
  the vignettes workflow will produce the new `examples.pdf`
  on next push.

### Why now

- The fixture corpus has grown from 9 (v0.2) to 35 (v0.5.2). The
  three single-purpose vignettes were authored when the corpus
  was small enough that each represented a meaningful slice.
  They are now a small minority of the validated material.
- v0.6.0's `payment_override` and `fee_per_period` features each
  produce *new* example types (FHLBB-style given-payment loans,
  fee-loaded annuities). Adding two more single-purpose vignettes
  on top of the three existing ones would produce navbar bloat;
  consolidating now and adding to a unified examples document
  is the cleaner path.
- The country-convention organization mirrors the way users
  actually approach the library: "I have a Canadian mortgage to
  model" or "what does the library do for ARM payment caps" both
  go to the same examples vignette and find the answer in a
  predictable section.
