# mortgagemath v0.6.1 - Promotion-readiness fixes

Status: **planning.** Branch (when started): `v0.6.1-promotion-readiness`.

## Scope

v0.6.1 should be a stabilization release before broader promotion.
It does not need a broad new mortgage feature. Its purpose is to
remove the review blockers found after v0.6.0 and make the public
documentation, Read the Docs build, and release artifacts trustworthy.

The review produced duplicate inline findings. This plan consolidates
them as follows:

| Review findings | Consolidated work item |
|---|---|
| 1 and 6 | Ambient Decimal context can change schedule output |
| 2 and 7 | `payment_override` can overpay into negative balances |
| 3 and 8 | `payment_override` validation is incomplete |
| 4 and 9 | Read the Docs install path is broken |
| 5 and 10 | Public validation counts and claims have drifted |

## Release gates

Before v0.6.1 is promoted, all of these commands should pass from a
clean checkout:

```sh
uv run --extra dev pytest
uv run --extra dev ruff check
uv run --extra dev mypy
uv run --extra docs sphinx-build -W -b html docs/sphinx docs/sphinx/_build/html
python -m mortgagemath
```

The final release candidate should also be built from a clean `dist/`
directory so no stale `0.5.x` or `0.6.x` artifacts can be uploaded by
mistake.

## 1. Schedule math must not depend on ambient Decimal precision

### Current failure

`periodic_payment()` protects the closed-form annuity formula with a
50-digit local Decimal context, but schedule generation performs raw
balance and interest arithmetic under the caller's global Decimal
context. A caller that lowers global precision can silently change
published fixture values.

Reproduction observed during review:

```python
from datetime import date
from decimal import Decimal, getcontext

from mortgagemath import (
    DayCount,
    LoanParams,
    PaymentRounding,
    amortization_schedule,
)

loan = LoanParams(
    principal=Decimal("25000000"),
    annual_rate=Decimal("5.5"),
    term_months=120,
    amortization_period_months=360,
    day_count=DayCount.ACTUAL_360,
    payment_rounding=PaymentRounding.ROUND_HALF_UP,
    interest_rounding=PaymentRounding.ROUND_HALF_UP,
    start_date=date(2018, 12, 1),
)

assert amortization_schedule(loan)[120].balance == Decimal("20885505.83")

getcontext().prec = 10

# Current buggy result observed: Decimal("20885505.23")
assert amortization_schedule(loan)[120].balance == Decimal("20885505.83")
```

### Target behavior

`amortization_schedule(loan)` should produce the same cent-rounded
schedule regardless of the caller's active Decimal precision or
rounding context, except for explicit rounding modes stored on
`LoanParams`.

### Implementation notes

- Wrap schedule generation in an explicit high-precision
  `localcontext()` using the same precision policy as
  `periodic_payment()` (`ctx.prec = 50` unless tests show a need for
  a higher internal precision).
- The public `amortization_schedule()` dispatcher is the best outer
  guard because it covers all schedule paths:
  - `_schedule_thirty_360_round_each`
  - `_schedule_thirty_360_carry_precision`
  - `_schedule_actual_360`
- It is still reasonable to add smaller local contexts around
  especially sensitive private helper math, but the public guard is
  the essential fix.
- Keep all `quantize(..., rounding=...)` calls explicit. The local
  context should provide precision, not replace loan-specific rounding
  choices.
- Be careful not to accidentally let helper functions mutate global
  context. Use `with localcontext() as ctx:` only.

### Tests

Add tests under `tests/test_schedule.py`:

1. A regression test for the Fannie Mae Actual/360 SARM fixture under
   low global precision:

   ```python
   def test_actual_360_independent_of_global_decimal_precision():
       original = decimal.getcontext().copy()
       try:
           decimal.getcontext().prec = 10
           sched = amortization_schedule(FANNIE_MAE_LOAN)
           assert sched[120].balance == Decimal("20885505.83")
       finally:
           decimal.setcontext(original)
   ```

2. A carry-precision 30/360 regression under low precision. Use an
   existing fixture-backed loan such as Geltner or Goldstein, and assert
   a published row still matches.

3. A default round-each 30/360 regression under low precision. This
   path is less sensitive but should be covered so all schedule paths
   are explicitly protected.

### Acceptance criteria

- The reproduction above passes.
- Existing 36 schedule fixtures still pass.
- Running the full suite with default Decimal context and with a
  deliberately low Decimal context around individual tests produces
  identical cent outputs.

## 2. `payment_override` must not overpay into impossible balances

### Current failure

The carry-precision schedule path treats `payment_override` as both
the displayed and raw payment, then subtracts it from the internal
balance without an early-payoff guard. A high override can drive the
loan balance below zero and later produce negative interest and a
negative final payment.

Reproduction observed during review:

```python
from decimal import Decimal

from mortgagemath import (
    BalanceTracking,
    LoanParams,
    amortization_schedule,
)

loan = LoanParams(
    principal=Decimal("1000"),
    annual_rate=Decimal("5"),
    term_months=12,
    balance_tracking=BalanceTracking.CARRY_PRECISION,
    payment_override=Decimal("500"),
)

sched = amortization_schedule(loan)

# Current buggy behavior observed:
# - row 3 balance is negative
# - later rows have negative interest
# - final payment is negative
assert all(row.balance >= Decimal("0.00") for row in sched)
assert sched[-1].payment >= Decimal("0.00")
```

### Target behavior

When a regular payment would retire the loan before the scheduled
final period:

- The schedule should truncate at the actual payoff period.
- The payoff row should true up the remaining balance plus interest.
- The final balance should be exactly `Decimal("0.00")`.
- No row caused by early payoff should show negative balance,
  negative interest, or negative payment.
- An `EarlyPayoffWarning` should be emitted, matching the existing
  round-each behavior for tiny-principal overpayment.

### Implementation notes

- Add an early-payoff guard to `_schedule_thirty_360_carry_precision`.
- The guard should run after `interest_raw` and `interest_disp` are
  computed and before the normal non-final balance update.
- For a fully amortizing loan, compute whether the current raw payment
  would pay off the loan:

  ```python
  will_pay_off_early = (
      not is_scheduled_final
      and fully_amortizing
      and pmt_raw - interest_raw >= balance
  )
  ```

- If early payoff is triggered:
  - Compute the payoff amount from full-precision `balance + interest_raw`.
  - Quantize it to cents with `payment_rounding`.
  - Compute displayed principal so the public invariant
    `principal + interest == payment` still holds.
  - Set internal and displayed balance to zero.
  - Append the payoff row, warn with `EarlyPayoffWarning`, and `break`.
- Preserve the FHLBB 1935 fixture behavior. Its normal final-row
  trueup must still match the published `29.27`.
- Consider applying the guard for all carry-precision loans, not only
  override loans. The bug is easiest to trigger with `payment_override`,
  but the invariant should be general.

### Tests

Add focused tests under `tests/test_schedule.py` or a new
`tests/test_payment_override.py`:

1. High override truncates early:

   ```python
   with pytest.warns(EarlyPayoffWarning):
       sched = amortization_schedule(high_override_loan)

   assert sched[-1].balance == Decimal("0.00")
   assert sched[-1].number < high_override_loan.term_months
   assert all(row.balance >= Decimal("0.00") for row in sched)
   assert all(row.payment >= Decimal("0.00") for row in sched)
   ```

2. Every row still satisfies `principal + interest == payment`.

3. FHLBB 1935 fixture still matches every committed CSV row.

4. A normal non-override carry-precision loan remains byte-identical to
   its previous fixture output.

### Acceptance criteria

- The high-override reproduction no longer produces negative balances
  or payments.
- Existing early-payoff tests for the round-each path still pass.
- FHLBB 1935 remains fixture-clean.

## 3. `payment_override` validation must happen before payment code

### Current failure

`periodic_payment()` returns `loan.payment_override` before validating
`amortization_period_months`. Meanwhile, `LoanParams.__post_init__`
only checks that the override is positive and incompatible with
`rate_schedule`. This means invalid combinations can reach payment and
schedule code:

- `payment_override` with a shorter amortization period can bypass the
  normal `amortization_period_months >= term_months` validation.
- `payment_override` with a balloon amortization basis is accepted even
  though the current comments describe override support as fixed-rate,
  fully amortizing only.
- Non-cent overrides such as `Decimal("99.999")` are accepted and can
  create public rows with non-cent payment amounts.

### Target behavior

`LoanParams` should reject unsupported override combinations at
construction time, before `periodic_payment()` or
`amortization_schedule()` sees them.

For v0.6.1, use conservative validation:

- `payment_override` must be positive.
- `payment_override` must be cent-denominated:

  ```python
  payment_override == payment_override.quantize(Decimal("0.01"))
  ```

- `payment_override` remains incompatible with non-empty
  `rate_schedule`.
- `payment_override` is supported only for fully amortizing loans:
  `amortization_period_months` must be either `None` or equal to
  `term_months`.
- General `amortization_period_months` validation should live in
  `LoanParams.__post_init__`, not only in `periodic_payment()`:
  - If set, it must be positive.
  - If set, it must be `>= term_months`.
  - It must pass the existing divisibility check for the selected
    payment frequency.

### Implementation notes

- Move the existing amortization-period positivity and
  `>= term_months` checks from `periodic_payment()` into
  `LoanParams.__post_init__`.
- Either remove the duplicate checks from `periodic_payment()` or keep
  them as defensive assertions. If kept, ensure they are not bypassed
  by the override return.
- Add a small `_PENNY = Decimal("0.01")` constant in `_types.py` if
  needed for cent validation.
- Consider rejecting non-finite Decimals (`NaN`, `Infinity`) for
  `principal`, `annual_rate`, and `payment_override` while touching
  validation. This is optional for v0.6.1 but cheap if done carefully.
- Preserve public error messages enough that existing tests do not
  fail unnecessarily, or update tests with clearer messages.

### Tests

Add validation tests:

```python
with pytest.raises(ValueError, match="payment_override"):
    LoanParams(
        principal=Decimal("1000"),
        annual_rate=Decimal("5"),
        term_months=12,
        payment_override=Decimal("99.999"),
    )

with pytest.raises(ValueError, match="fully amortizing|amortization"):
    LoanParams(
        principal=Decimal("1000"),
        annual_rate=Decimal("5"),
        term_months=120,
        amortization_period_months=360,
        payment_override=Decimal("100.00"),
    )

with pytest.raises(ValueError, match="must be >= term_months"):
    LoanParams(
        principal=Decimal("1000"),
        annual_rate=Decimal("5"),
        term_months=12,
        amortization_period_months=6,
        payment_override=Decimal("100.00"),
    )
```

Add regression coverage that `periodic_payment()` no longer returns an
override for an invalid loan object. Ideally the invalid object cannot
be constructed at all.

### Acceptance criteria

- Invalid override combinations fail in `LoanParams.__post_init__`.
- `periodic_payment()` can no longer bypass amortization validation.
- Existing FHLBB override fixture remains valid.

## 4. Read the Docs config and docs build must be reliable

### Current failure

`.readthedocs.yaml` currently contains:

```yaml
path: .I
```

This should be:

```yaml
path: .
```

The review also found that a strict local Sphinx build currently exits
nonzero because warnings are emitted.

### Target behavior

Read the Docs should install the package from the repository root and
the local docs build should be clean enough that CI can run
`sphinx-build -W`.

### Implementation notes

1. Fix `.readthedocs.yaml`:

   ```yaml
   python:
     install:
       - method: pip
         path: .
         extra_requirements:
           - docs
   ```

2. Add a docs build check to CI or document it as a release gate. The
   command should be:

   ```sh
   uv run --extra docs sphinx-build -W -b html docs/sphinx docs/sphinx/_build/html
   ```

3. Fix the known Sphinx warnings:
   - `docs/sphinx/api.md` currently indexes objects more than once
     because it has an `automodule` member listing plus individual
     `autofunction` / `autoclass` directives. Choose one indexing path:
     either remove `:members:` from `automodule`, or add `:no-index:` to
     the duplicate entries.
   - `docs/sphinx/changelog.md` has a MyST include option that Sphinx
     reports as malformed. Quote the `start-after` value or simplify the
     include so it no longer warns.
   - Fix any missing cross-reference emitted from included
     `CHANGELOG.md` content, especially references to `docs/accuracy.md`
     from inside the Sphinx tree.

### Tests

- Run the Sphinx command above locally.
- If CI is updated, confirm the docs job runs on pull requests.
- Confirm Read the Docs builds after the config fix.

### Acceptance criteria

- `.readthedocs.yaml` points at `path: .`.
- `sphinx-build -W` exits 0.
- The hosted Read the Docs build succeeds for the v0.6.1 branch or
  release tag.

## 5. Public validation counts and claims must not drift

### Current failure

The current public docs disagree with each other:

- `README.md` says 37 fixtures.
- `docs/sphinx/index.md` says 27 fixtures.
- `docs/sphinx/vignettes.md` says 36 fixtures.
- The repository currently contains 36 paired TOML/CSV schedule
  fixtures.

There are also stale statements elsewhere:

- `docs/vignettes/history.qmd` says the FHLBB 1935 given-payment
  convention is not modeled, but v0.6.0 introduced
  `payment_override` and ships the FHLBB Plan A fixture.
- `docs/future-work.md` says Canadian semi-annual compounding is not
  implemented, but v0.3.0+ supports `Compounding.SEMI_ANNUAL`.
- Marketing copy says every published cell matches, but some sources
  have documented typos or excluded inconsistent cells. The more
  accurate claim is that every committed fixture cell matches the
  source value selected for validation, with known source errors
  documented rather than forced into the fixture.

### Target behavior

All public docs should give one consistent, defensible validation
claim:

- The fixture count should match the actual paired TOML/CSV fixture
  count.
- Claims should distinguish "committed fixture cells match" from
  "every cell in every source document matches" when known source
  errors were excluded.
- Stale future-work/history claims should be removed or rewritten.

### Implementation notes

1. Decide whether fixture count is 36 or whether another fixture should
   land before v0.6.1. Then update every public location consistently:
   - `README.md`
   - `docs/sphinx/index.md`
   - `docs/sphinx/vignettes.md`
   - rendered Quarto sources that print the count
   - `docs/vignettes/history.qmd`
   - `CHANGELOG.md`, if it repeats the count

2. Prefer centralizing the count instead of hand-editing multiple
   numbers. Reasonable options:
   - Generate the validation count from fixture TOML metadata during
     Quarto rendering and Sphinx docs build.
   - Add a small test that computes the fixture count and asserts that
     key docs do not contain stale hard-coded counts.
   - Add a simple script used by release docs generation to update a
     small include file such as `docs/generated/fixture-count.md`.

3. Recommended wording for README-style claims:

   ```md
   36 paired TOML/CSV fixtures validate published regulatory, GSE,
   historical, and textbook examples. Every committed fixture row
   matches its source value to the cent; known source arithmetic typos
   are documented rather than forced into the fixture corpus.
   ```

4. Update the stale FHLBB history section:
   - Remove "Given-payment, find-term contracts are not modeled."
   - Replace it with a note that v0.6.0 models the payment-pinned
     direct-reduction convention via `payment_override`, validated
     against FHLBB 1935 Plan A.

5. Update the stale Canadian future-work section:
   - Remove the statement that semi-annual compounding is missing.
   - Keep any still-unmatched Canadian examples only if their remaining
     mismatch is due to a different issue, such as missing-penny
     reconciliation or non-actuarial whole-dollar payments.

### Tests

Add a lightweight documentation consistency test. For example:

```python
from pathlib import Path


def test_public_fixture_count_claims_are_current():
    fixture_count = len(
        [
            p
            for p in Path("tests/schedules").glob("*.toml")
            if p.with_suffix(".csv").exists()
        ]
    )
    assert fixture_count == 36

    readme = Path("README.md").read_text()
    assert f"{fixture_count} paired TOML/CSV fixtures" in readme
```

If that is too brittle, at least add a release checklist item requiring
the fixture count to be regenerated before publishing.

### Acceptance criteria

- README, Sphinx docs, Quarto docs, and changelog agree on fixture
  count.
- Stale FHLBB and Canadian claims are corrected.
- The "every published cell" language is softened to the defensible
  "every committed fixture cell" claim wherever needed.

## Suggested implementation order

1. Fix `.readthedocs.yaml` and Sphinx warnings first. This gives a
   clean docs gate for the rest of the work.
2. Move amortization-period and override validation into
   `LoanParams.__post_init__`.
3. Add the carry-precision early-payoff guard for overlarge
   `payment_override`.
4. Wrap schedule generation in an explicit local Decimal context and
   add low-precision regression tests.
5. Sweep docs claims and fixture counts.
6. Remove or rebuild stale release artifacts in `dist/` and
   `dist-pypi/`.
7. Run the full release gate command list from this plan.

## Out of scope for v0.6.1

The broader product recommendations from the review are valuable but
should not block this stabilization release:

- Easy-mode constructors such as `fixed_payment(...)`.
- Extra principal payments and recast strategies.
- Indexed ARM builders.
- Fees/APR/effective-rate support.
- Zero-interest loans.
- Constant-principal amortization.
- Multi-tranche loans.
- Country presets and European fixture expansion.

Those belong in v0.8.0+ after the v0.6.1 promotion-readiness fixes
are complete.
