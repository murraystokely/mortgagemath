# mortgagemath v0.5.0 — Payment caps + negative amortization

Status: **implemented, awaiting commit and release.**  Branch:
`v0.5.0-payment-caps`.

## Scope

This release ships exactly one new library capability:

- **Payment caps with negative amortization** on Adjustable-Rate
  Mortgages.  Extends ``RateChange`` with an optional
  ``payment_cap_factor`` field that bounds the per-reset payment
  increase relative to the prior period's payment.  When the cap binds
  *and* periodic interest exceeds the capped payment, the unpaid
  interest is capitalized into the balance (negative amortization).

Validated against **ProEducate ARM Payment Caps worked example**
($65,000 / Year 1 at 10% / Year 2 at 12% / 30yr / 7.5% payment
cap) — every published anchor reproduces to the cent under
``BalanceTracking.ROUND_EACH`` + ``ROUND_HALF_UP``:

| Anchor | Published | Reason |
|---|---|---|
| Year 1 monthly P&I | $570.42 | Closed-form annuity at 10% |
| Balance after pmt 12 | $64,638.72 | Row-by-row roll-forward |
| Year 2 uncapped recast | $667.30 | Closed-form on remaining 348 mo at 12% |
| Year 2 capped P&I | $613.20 | $570.42 × 1.075 |
| Cumulative neg-am Y2 | $420.90 | Sum of (interest − $613.20) for 12 mo |
| Balance after pmt 24 | $65,059.62 | $64,638.72 + $420.90 |

Verified via a Decimal simulation matching the proposed library
algorithm before any code was written.

## API design

### `RateChange.payment_cap_factor`

```python
@dataclass(frozen=True, slots=True)
class RateChange:
    effective_payment_number: int
    new_annual_rate: Decimal
    recast: bool = True
    payment_cap_factor: Decimal | None = None   # ← new in v0.5
```

When ``recast=True`` and ``payment_cap_factor`` is set:

```
uncapped = closed-form annuity payment at new rate over remaining payments
          (computed against the current balance)
cap      = prior_period_payment * payment_cap_factor   (cents-rounded)
new_payment = min(uncapped, cap)
```

If ``cap_factor=None``, the recast computes the unconstrained closed-
form payment (existing v0.4 behavior).  If ``recast=False``, the prior
payment carries forward unchanged regardless of ``cap_factor``.

### Validation (in ``RateChange.__post_init__``)

- ``payment_cap_factor`` (when set) must be > 0.  Setting it to a
  value < 1 is allowed (the cap can constrain *down* as well as up).
- All existing v0.4 invariants preserved.

### Negative amortization in `Installment`

Pre-v0.5: ``Installment.principal`` is always positive (the portion
of the payment that reduces principal).  Per-row invariant
``principal + interest == payment`` always held trivially.

With v0.5 payment caps: when ``interest > payment`` for a row, the
unpaid interest is capitalized into the balance.  This shows up as
**negative ``Installment.principal``** with
``balance_new = balance_old - principal = balance_old + |principal|``
(balance grows).  The invariant ``principal + interest == payment``
still holds.

Documented in ``Installment``'s docstring as a permitted v0.5
behavior; a ``NEG_AM`` warning category is **not** added — neg-am is
intended behavior under a payment cap, not an exceptional condition.

## Schedule generation — algorithmic change

In both ``_schedule_thirty_360_round_each`` and the carry-precision
sibling, the rate-change branch grows an additional step:

```python
if rc is not None:
    new_rate = _periodic_rate_for(rc.new_annual_rate, ...)
    if rc.recast:
        remaining = total_payments - (i - 1)
        pmt_uncapped_raw, pmt_uncapped = _recast_payment_pair(
            balance, new_rate, remaining, loan.payment_rounding
        )
        if rc.payment_cap_factor is not None:
            cap = (pmt * rc.payment_cap_factor).quantize(_PENNY, rounding=…)
            pmt = min(pmt_uncapped, cap)
        else:
            pmt = pmt_uncapped
    periodic_rate = new_rate
    rate_schedule_idx += 1
```

The per-row body is unchanged — ``interest = balance * periodic_rate``
and ``principal = pmt - interest``.  The only thing that changes is
that ``principal`` may now be negative.

## CLI extension

The ``--rate-change`` flag's syntax extends to support cap factor:

```
EFFECTIVE_PMT:NEW_RATE                   # unchanged (no cap)
EFFECTIVE_PMT:NEW_RATE:no_recast         # unchanged
EFFECTIVE_PMT:NEW_RATE:cap=FACTOR        # new — caps recast at FACTOR
EFFECTIVE_PMT:NEW_RATE:no_recast:cap=FACTOR   # disallowed (cap requires recast)
```

The third form uses ``cap=`` as a key=value suffix to disambiguate
from the bare ``no_recast`` flag.  Validation: ``cap=`` is only
allowed when recast is true.

## Goldstein §10.4 Example 14 — investigated, NOT shipped

Goldstein's $200k / 5/1 ARM example is the canonical pedagogical
source for ARM payment caps + negative amortization.  Its year-7
mechanics (uncapped $1,625, capped $1,440.88 at 8% × prior $1,334.15,
month-73 neg-am $20.35) match the ProEducate algorithm exactly
*conceptually*.

But Goldstein computes the year-6 balance ($185,405.12) via the
closed-form outstanding-balance formula at the rounded year-1 payment
$1,160.80 — `pmt × (1 - (1+r)^-(n-k))/r` — whereas the library's
row-by-row schedule produces $185,405.38 (round-each) or $185,405.25
(carry-precision), a 13-26¢ drift at month 60.  This propagates to
the year-6 P&I ($1,334.16 vs $1,334.15) and breaks the year-7 cap
calculation by 1¢ ($1,440.89 vs $1,440.88, since 1334.16 × 1.08 =
1440.89).

Same algorithmic mismatch documented for Goldstein §10.4 Ex 13 in
``docs/v0.4-plan.md``.  Per the no-partial-fixtures rule, Goldstein
Ex 14 is NOT shipped — but is documented in ``CHANGELOG.md`` as the
pedagogical motivation for the API.  ProEducate is the published-
source fixture that validates the cents-level correctness.
